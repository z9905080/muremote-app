# MuRemote App - POC 產品規格文件

## 1. 產品定位

| 項目 | 內容 |
|------|------|
| 產品名稱 | MuRemote App |
| 類型 | 遠端控制軟體 (SaaS) |
| 目標模擬器 | MuMu 模擬器 (網易) |
| 競爭對手 | OSLink (綁定雷電模擬器) |
| 核心差異點 | 支援 MuMu 模擬器，不限定單一模擬器 |

---

## 2. 市場背景

### 痛點
- OSLink 僅支援雷電模擬器，不支援 MuMu
- MuMu 用戶有遠端控制需求（掛機、工作室、手機操作電腦模擬器）
- 現有遠端軟體（TeamViewer、AnyDesk）不針對模擬器優化

### 機會
- MuMu 模擬器用戶持續成長
- 遊戲工作室需要多開管理
- 手機遠端操作場景需求增加

## 4. 技術架構

### 系統架構圖
```
┌─────────────┐      WebRTC/RTSP      ┌─────────────┐      ADB       ┌─────────────┐
│  手機 App   │ ◄──────────────────► │   PC Client │ ◄───────────► │ MuMu 模擬器 │
│ (Flutter)   │                       │ (Electron)  │               │             │
└─────────────┘                       └─────────────┘               └─────────────┘
```

### 技術選型

| 層面 | 技術 |
|------|------|
| 手機端 | Flutter |
| PC 客戶端 | Electron / Tauri |
| 螢幕擷取 | ADB + Scrcpy 協議 |
| 串流傳輸 | WebRTC / RTSP |
| 遠端控制 | ADB input events |
| 帳號系統 | Firebase Auth / 自建 |

### 數據流
1. **螢幕串流**: MuMu → ADB → PC Client → 編碼 → WebRTC → 手機 App
2. **操控回傳**: 手機觸控 → 映射邏輯 → ADB input → MuMu 模擬器

## 5. 驗證指標 (POC Success Criteria)

| 指標 | 目標值 | 備註 |
|------|--------|------|
| 延遲 | < 200ms | 區域網路環境 |
| 幀率 | 30 fps | MVP 標準 |

## 7. 風險與對策

| 風險 | 影響 | 對策 |
|------|------|------|
| MuMu API 不開放 | 無法直接控制 | 使用 ADB 通用協議 |
| 延遲過高 | 體驗差 | 優化編碼參數、支援區域網路優先 |
| 被判定作弊 | 遊戲封號 | 僅做操作工具，不修改遊戲記憶體 |
